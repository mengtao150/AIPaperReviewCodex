# 综述稿件审查系统需求文档与技术实现路径

## 1. 项目背景

本系统用于审查学术综述类稿件。用户在网页端上传稿件后，系统先让大模型理解 Review 类型判别图，再让大模型阅读整篇文章并判断该稿件属于哪一种综述类型。类型确定后，系统从模板库中匹配对应的文章结构模板，并从清单库中匹配对应的具体审核清单，最后由大模型逐项判断稿件是否按要求撰写，并输出缺失项和修改建议。

当前项目中的初始规则来源包括：

- `docx/文章结构.docx`：通用文章结构要求。
- `docx/综述清单.pdf`：Review 类型判别图，以及 Narrative Review、Mini Review、Clinical Practice Review 等类型的具体审核要求示例。

后续系统需要支持更多综述类型、更多 Word 模板、更多审核清单，因此模板库和清单库必须可配置、可扩展、可版本管理。

## 2. 核心目标

建设一个网页端综述审查系统，实现以下能力：

1. 上传综述稿件。
2. 让大模型学习并理解 Review 类型判别图。
3. 让大模型阅读全文，并按照判别图逻辑判断稿件所属综述类型。
4. 根据判别出的综述类型，在文章结构模板库中匹配对应 Word 模板。
5. 审查稿件是否按照该类型的文章结构模板撰写。
6. 根据判别出的综述类型，在审核清单库中匹配对应清单。
7. 让大模型阅读审核清单和稿件全文，逐项判断具体内容是否已撰写。
8. 对未撰写、不完整、无法判断的项目，列出原因和修改建议。
9. 输出完整审查报告。

## 3. 总体业务流程

系统主流程如下：

1. 用户上传稿件。
2. 系统解析稿件内容，包括题目、摘要、关键词、章节、正文、图表说明、参考文献等。
3. 系统把 Review 类型判别图转换为大模型可理解的判别准则。
4. 大模型阅读判别准则和稿件全文，判断稿件属于哪一种综述类型。
5. 系统根据判别结果匹配文章结构模板。
6. 系统和大模型共同检查稿件是否符合该结构模板。
7. 系统根据判别结果匹配该综述类型的具体审核清单。
8. 大模型阅读审核清单和稿件全文，逐项判断是否撰写到位。
9. 系统汇总结构审查和清单审查结果，生成审查报告。

## 4. Review 类型判别逻辑

类型判别依据来自 `综述清单.pdf` 中的 Review 类型判别图。系统应将其整理为模型提示词和结构化规则。

判别逻辑如下：

1. 如果文献检索系统，并且纳入了系统评价类研究，则判断为 `Overview of Reviews`。
2. 如果文献检索系统、全面，并且对纳入研究进行了质量评估，例如偏倚风险评估，则判断为 `Scoping Review`。
3. 如果文献检索系统、全面，但未对纳入研究进行质量评估，则判断为 `Systematic Review`。
4. 如果文献检索有时间限制，则判断为 `Rapid Review`。
5. 如果文章字数较少，则判断为 `Mini Review`。
6. 如果文章是基于专家观点、意见或经验的文献讨论，则判断为 `Clinical Practice Review`。
7. 如果以上均不符合，则判断为 `Narrative Review`。

重要要求：

- 大模型不能只根据题目判断类型。
- 大模型必须阅读全文，尤其关注摘要、方法或检索策略、主体内容和结论。
- 大模型需要输出判别路径，而不仅是最终类型。
- 如果判断证据不足，应标记为需要人工确认。

## 5. 功能需求

### 5.1 稿件上传与解析

第一版优先支持 `.docx` 稿件上传。PDF 稿件上传可以作为后续能力。

系统需要从稿件中提取：

- 题目
- 作者和机构信息
- 摘要
- 关键词
- 一级、二级、三级标题
- 正文内容
- 表格内容
- 图题和图注
- 参考文献

解析结果需要保留章节顺序，便于后续判断结构是否完整、顺序是否合理。

### 5.2 大模型类型判别

系统将以下内容发送给大模型：

- Review 类型判别准则
- 稿件全文解析结果
- 输出 JSON 格式要求

模型返回结果应包括：

- 最终综述类型
- 置信度
- 判别路径
- 支持该判断的原文证据
- 存疑点
- 是否建议人工确认

示例输出字段：

- `review_type`
- `confidence`
- `decision_path`
- `evidence`
- `uncertainties`
- `needs_human_review`

### 5.3 文章结构模板库

文章结构模板库用于保存每一种综述类型对应的 Word 结构模板。

每个结构模板应包含：

- 综述类型
- 模板名称
- 模板版本
- 原始 Word 模板文件
- 必需章节
- 可选章节
- 推荐章节顺序
- 特殊说明
- 是否启用

第一版可以先把 `文章结构.docx` 解析为通用结构模板。后续再支持为 Narrative Review、Mini Review、Clinical Practice Review 等类型分别上传独立 Word 模板。

### 5.4 文章结构审查

系统根据已判别出的综述类型，选择对应结构模板，并检查稿件是否符合结构要求。

每个结构项目应输出以下状态之一：

- 已撰写
- 未撰写
- 撰写不完整
- 使用了不同标题但内容存在
- 不适用
- 需要人工确认

结构审查需要给出证据，例如匹配到的标题、段落位置或相关原文。

### 5.5 审核清单库

审核清单库用于保存每种综述类型的具体内容要求。它与文章结构模板库分开管理。

每个清单模板应包含：

- 综述类型
- 清单名称
- 清单版本
- 来源文件
- 审查维度
- 审查项目
- 是否必需
- 评价说明
- 排序
- 是否启用

第一版先从 `综述清单.pdf` 中提取已有示例，包括：

- Narrative Review 审核清单
- Mini Review 审核清单
- Clinical Practice Review 审核清单

后续建立独立清单库，支持在网页端新增、编辑、导入和版本管理。

### 5.6 审核清单逐项审查

系统根据综述类型匹配对应审核清单，然后让大模型阅读：

- 稿件全文
- 对应综述类型的审核清单
- 每条清单的评价要求

大模型需要逐项判断清单内容是否在稿件中撰写。

每个清单项目应输出：

- 状态：已撰写、未撰写、撰写不完整、无法判断、不适用
- 原文证据
- 判断理由
- 修改建议
- 建议补写位置

例如 Narrative Review 清单中要求报告检索日期、检索时间范围、数据库、检索词和纳排标准时，模型需要判断稿件是否具体写出这些内容，而不是只出现笼统表述。

### 5.7 审查报告

审查报告应包含：

- 稿件名称
- 判别出的综述类型
- 类型判别依据
- 匹配到的文章结构模板
- 结构审查结果
- 匹配到的审核清单
- 清单逐项审查结果
- 未撰写项目汇总
- 撰写不完整项目汇总
- 修改建议
- 需要人工确认的项目

第一版可以在网页端展示并导出 Markdown。后续可支持导出 Word 审查报告。

## 6. 非功能需求

### 6.1 可配置性

新增综述类型、结构模板、审核清单时，不应修改代码。管理员应能通过模板库和清单库维护规则。

### 6.2 可追溯性

每个判断结果都需要保留依据：

- 使用了哪条判别规则
- 匹配了哪个模板
- 匹配了哪条审核清单
- 稿件中的哪段内容支持该判断

### 6.3 人工复核

系统不能把不确定判断包装成确定结论。类型判别或清单匹配存在疑问时，应明确标记为需要人工确认。

### 6.4 版本管理

结构模板和审核清单都需要版本号。后续如果期刊要求或审查标准更新，历史审查结果仍能追溯当时使用的模板版本。

### 6.5 隐私与文件管理

第一版建议本地存储上传文件。后续如部署到服务器，需要增加：

- 用户权限
- 文件访问控制
- 文件保留期限
- 删除机制
- 日志脱敏

## 7. 推荐页面设计

### 7.1 稿件审查页面

用于完成主要审查流程：

1. 上传稿件。
2. 点击开始审查。
3. 查看审查进度。
4. 查看最终报告。

进度状态包括：

- 文件解析中
- 类型判别中
- 结构模板匹配中
- 结构审查中
- 审核清单匹配中
- 清单逐项审查中
- 报告生成中
- 审查完成

### 7.2 文章结构模板库页面

用于管理不同综述类型的文章结构模板：

- 上传 Word 模板
- 编辑模板名称和版本
- 查看解析出的章节结构
- 绑定综述类型
- 启用或停用模板

### 7.3 审核清单库页面

用于管理不同综述类型的具体审核清单：

- 新增清单
- 导入清单
- 编辑清单项目
- 绑定综述类型
- 设置必需项和可选项
- 管理清单版本

### 7.4 审查报告页面

用于查看审查结果：

- 总览
- 类型判别依据
- 结构审查
- 清单审查
- 缺失项
- 修改建议
- 人工确认项

报告页面应支持筛选：

- 只看未撰写
- 只看撰写不完整
- 只看无法判断
- 只看需要人工确认

## 9. 技术实现路径

### 9.1 推荐技术栈

建议采用简单清晰的前后端架构：

- 前端：React 或 Next.js
- 后端：Python FastAPI
- 数据库：第一版使用 SQLite，后续生产环境使用 PostgreSQL
- Word 解析：`python-docx`
- PDF 解析：`pdfplumber`
- 大模型接口：OpenAI API 或可配置模型提供商

选择 Python 后端的原因是文档解析、PDF 处理、模型编排和报告生成更方便。前端负责上传、配置和结果展示。

### 9.2 后端模块划分

后端建议拆成以下服务模块：

1. `DocumentExtractor`
   - 解析上传的 `.docx` 稿件。
   - 提取标题、摘要、章节、正文、表格、图注和参考文献。

2. `ClassificationRuleProvider`
   - 保存 Review 类型判别图对应的规则。
   - 输出模型可理解的判别准则。

3. `ReviewTypeClassifier`
   - 调用大模型。
   - 输入稿件全文和判别准则。
   - 输出综述类型、置信度、判别路径和证据。

4. `TemplateMatcher`
   - 根据综述类型匹配启用中的文章结构模板。

5. `StructureAuditor`
   - 根据模板要求检查稿件结构。
   - 输出结构项目的审查状态和证据。

6. `ChecklistMatcher`
   - 根据综述类型匹配启用中的审核清单。

7. `ChecklistAuditor`
   - 调用大模型逐项检查清单要求。
   - 输出每条清单的状态、证据、理由和修改建议。

8. `ReportBuilder`
   - 汇总类型判别、结构审查和清单审查。
   - 生成最终报告。

### 9.3 大模型调用策略

建议所有大模型任务都要求输出 JSON，便于系统稳定解析。

类型判别 Prompt 应包含：

- Review 类型判别准则
- 稿件全文
- 必须按全文判断的要求
- 输出 JSON schema
- 必须提供证据的要求
- 存疑时标记人工确认的要求

清单审查 Prompt 应包含：

- 已判别出的综述类型
- 匹配到的审核清单
- 稿件全文
- 每项都必须独立判断的要求
- 未撰写或不完整时必须提出修改建议的要求
- 输出 JSON schema

结构审查可以先用规则匹配标题和章节，再让大模型复核标题不同但内容存在的情况。

### 9.4 初始数据导入

第一版从现有文件建立种子数据：

1. 从 `文章结构.docx` 提取通用文章结构。
2. 从 `综述清单.pdf` 提取 Review 类型判别规则。
3. 从 `综述清单.pdf` 提取 Narrative Review 审核清单。
4. 从 `综述清单.pdf` 提取 Mini Review 审核清单。
5. 从 `综述清单.pdf` 提取 Clinical Practice Review 审核清单。
6. 将这些内容保存到数据库或本地 seed JSON 文件中。

由于 PDF 包含流程图和复杂表格，第一版允许人工校对导入结果，保证规则准确。

### 9.5 错误处理

系统需要处理以下情况：

- 文件格式不支持。
- 文档解析失败。
- 大模型类型判别置信度低。
- 未找到对应结构模板。
- 未找到对应审核清单。
- 大模型接口调用失败。
- 大模型输出不是合法 JSON。
- 稿件内容过长，需要分段处理。

如果缺少结构模板或审核清单，系统仍应输出已完成的部分，并清楚说明哪些环节无法审查。

### 9.6 测试策略

第一版至少需要以下测试：

- `.docx` 解析单元测试。
- 类型判别结果 JSON 解析测试。
- 结构模板匹配测试。
- 审核清单匹配测试。
- 报告生成快照测试。
- 模拟大模型返回结果的端到端测试。
- 使用一篇样例稿件跑完整流程的集成测试。

## 10. 分阶段交付计划

### Phase 1：本地原型

目标是跑通完整审查链路。

交付内容：

- 网页上传稿件。
- `.docx` 内容解析。
- 固化 Review 类型判别图规则。
- 大模型判断综述类型。
- 从现有文件导入结构模板和清单。
- 匹配结构模板。
- 匹配审核清单。
- 大模型逐项清单审查。
- 网页展示审查报告。
- Markdown 报告导出。

### Phase 2：模板库和清单库

目标是让系统可维护、可扩展。

交付内容：

- 结构模板库页面。
- 审核清单库页面。
- 上传 Word 模板。
- 解析模板内容。
- 手动编辑解析结果。
- 绑定综述类型。
- 模板和清单版本管理。

### Phase 3：审查报告增强

目标是让结果更适合实际编辑审稿工作。

交付内容：

- 富文本报告页面。
- 缺失项和不完整项筛选。
- 人工确认项标记。
- 审查历史。
- Word 报告导出。
- 修改建议汇总。

### Phase 4：生产化

目标是支持多人和长期使用。

交付内容：

- 用户账号。
- 权限管理。
- PostgreSQL 数据库。
- 后台任务队列。
- 文件保留和删除策略。
- 模型提供商配置。
- 部署脚本。

## 11. 待确认事项

以下事项不影响第一版设计，但会影响实现优先级：

1. 第一版是否必须支持 PDF 稿件上传，还是先只支持 `.docx`。
2. 第一版是否必须导出 Word 审查报告，还是先导出 Markdown。
3. 使用哪个大模型提供商和具体模型。
4. 上传稿件是否长期保留在审查历史中。
5. 是否需要用户登录和权限控制。

## 12. 第一版成功标准

第一版完成后，应能做到：

1. 用户上传一篇 `.docx` 综述稿件。
2. 系统解析出稿件主要内容。
3. 大模型按照 Review 类型判别图判断综述类型。
4. 系统根据类型匹配文章结构模板。
5. 系统根据类型匹配具体审核清单。
6. 大模型逐项判断清单要求是否在稿件中撰写。
7. 审查报告清楚列出已撰写、未撰写、撰写不完整和无法判断的项目。
8. 未撰写和撰写不完整项目均给出可执行修改建议。

## 13. 已确认技术栈修订

经确认，第一版实现采用以下技术路线：

- 前端：Vue 3 + Element Plus。
- 后端：Spring Boot。
- 数据库：MySQL。
- Word 解析：Apache POI。
- PDF 解析：Apache PDFBox，后续也可补充更强的 PDF 表格解析方案。
- 大模型：通过后台配置多供应商，支持填写 `Base URL`、`API Key`、`Model Name`、`Temperature`、`Max Tokens`、`Timeout` 等参数。
- 大模型接口优先支持 OpenAI-compatible 格式，后续可以通过适配器扩展其他非兼容接口。

第一版代码目录约定：

- `backend/`：Spring Boot 后端。
- `frontend/`：Vue 3 + Element Plus 前端。
